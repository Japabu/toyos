# bootstrap-cc

Bootstraps TinyCC on the host in two stages, with the end goal of replacing stage 1 with
toyos-cc — producing a TCC binary compiled entirely by our own compiler, which then compiles
itself.

**Target platforms: macOS, ToyOS.**

## Stages

1. **Stage 1** — toyos-cc compiles `tinycc/tcc.c` → `tcc-stage1`
2. **Stage 2** — `tcc-stage1` compiles `tinycc/tcc.c` → `tcc-stage2` (TCC self-hosted)

## End goal

When toyos-cc is capable enough:

1. toyos-cc compiles TCC → `tcc-stage1`
2. `tcc-stage1` compiles TCC → `tcc-stage2`
3. Both stages work on macOS and ToyOS

This validates toyos-cc against a real-world C codebase and produces a self-hosted TCC binary
without any external C compiler dependency.

## Key insights

### Target is the host

TCC is a native compiler. It compiles and links for the machine it runs on. The target defines
must match the host — on macOS arm64:

```
-DTCC_TARGET_ARM64
-DTCC_TARGET_MACHO
-DCONFIG_NEW_MACHO=1     # macOS 11+
```

No cross-compilation is involved. Both stages run on and produce binaries for the host.

### CONFIG_TCC_SEMLOCK=0 is required on macOS

If unset, `tcc.h` defaults `CONFIG_TCC_SEMLOCK` to 1, which pulls in `<dispatch/dispatch.h>`.
That header uses C extensions toyos-cc (and other simple compilers) can't parse. Defining it
as 0 satisfies `#ifdef` guards in `tcc.h` without triggering the GCD include.

### config.h and tccdefs_.h must be generated before compiling

`./configure` normally produces these. We generate `config.h` manually:

```c
#define CONFIG_TCC_PREDEFS 1  // embed tccdefs_.h string into the binary
#define GCC_MAJOR 4
#define GCC_MINOR 0
```

`CONFIG_TCC_PREDEFS=1` causes TCC to embed a stringified `include/tccdefs.h` directly into its
binary. `tccdefs_.h` is generated by compiling `conftest.c -DC2STR` and running it on
`include/tccdefs.h`. Without it TCC cannot find its built-in type definitions at runtime.

### libtcc1.a must be built with stage 1 before linking stage 2

TCC needs `libtcc1.a` — a small runtime with helpers for arithmetic and ABI support. It must
be compiled with stage 1 TCC, not the host compiler, because TCC's object format may differ.

On arm64: build from `lib/libtcc1.c` + `lib/lib-arm64.c`. The archive must live in
`CONFIG_TCCDIR`.

### TCCDIR is not automatically on the linker search path

`tcc_lib_path` (set from `CONFIG_TCCDIR`) is used to expand `{B}` in include paths but is
*not* added to `library_paths` — the list the linker searches for `libtcc1.a`. Pass `-L$TCCDIR`
explicitly when linking stage 2.

### TCC-produced objects cannot be linked by the host linker

TCC's object files are not consumable by clang's `ld` on macOS. Both the compile and link steps
for stage 2 must be driven by TCC. Don't compile with TCC (`-c`) and link with `cc`.

### macOS requires ad-hoc codesigning for TCC-linked binaries

Binaries linked by TCC on macOS are not codesigned and will be SIGKILLed on execution:

```
codesign --sign - <binary>
```

This applies to every executable TCC produces, including stage 2 itself.
